Product Requirements Document

UOS TV — Migrasi Base OS dari Alpine ke Armbian/Debian

Dokumen PRD lengkap untuk membangun image UOS TV berbasis Armbian yang lebih stabil di perangkat Amlogic S905X seperti HG680P dan B860H, dengan fokus pada boot reliability, kemudahan driver, browser/UI, audio, dan CI/CD yang reproducible.

Target hardware: HG680P / B860H / S905X Base: Armbian / Debian Kernel proven: 6.12.91-ophub Boot: aml_autoscript → uEnv.txt → booti Status: PRD awal

1. Ringkasan Eksekutif

Rekomendasi utama: UOS TV sebaiknya memakai Armbian/Debian sebagai base OS untuk fase produk awal, bukan Alpine/custom rootfs. Ini akan mempercepat bootable image, mengurangi risiko kernel/module mismatch, mempermudah audio/browser/WiFi, dan menyederhanakan CI/CD.

Eksperimen sebelumnya menunjukkan bahwa image UOS berbasis Alpine dapat berjalan bila menggunakan kernel, initrd, dan DTB dari Armbian. Kernel mainline self-built 6.6 stuck di splash screen pada hardware, sedangkan Armbian 6.12.91-ophub sudah terbukti boot di HG680P/B860H melalui mekanisme stock Amlogic u-boot.

Karena target produk adalah TV OS yang butuh UI, browser runtime, audio, input remote, WiFi, dan OTA, Debian/Armbian memberikan fondasi lebih praktis: paket tersedia, dependency native lebih mudah, systemd siap pakai, dan driver stack lebih matang.

2. Latar Belakang

2.1 Kondisi Saat Ini

  • Repo: https://github.com/subekti404dev/uos-tv
  • Workflow CI: .github/workflows/s905x-build.yml
  • Target image awal: Alpine rootfs + UOS services + kernel self-built + SD image.
  • Bootloader approach terbaru: Armbian-style, tanpa FIP, mengandalkan stock u-boot dan aml_autoscript.

2.3 Masalah Alpine sebagai Base

  • Musl toolchain membuat dependency native seperti ALSA lebih sulit.
  • Browser runtime seperti WPE WebKit/Cog/Chromium lebih rumit dibanding Debian.
  • Kernel module compatibility harus dijaga manual.
  • Custom supervision membuat boot loop mudah terjadi bila binary/service hilang.

3. Tujuan dan Non-Tujuan

Tujuan

  • Menghasilkan SD card image UOS TV yang bootable dan repeatable melalui GitHub Actions.
  • Memakai Armbian/Debian rootfs sebagai base utama.
  • Mempertahankan boot flow yang sudah proven: stock u-boot → aml_autoscript → uEnv.txt → kernel.
  • Menyediakan UI TV shell yang auto-start.
  • Mendukung network, WiFi, audio, input remote/keyboard, logging, dan recovery dasar.
  • Menghasilkan artifact .img yang kompatibel Balena Etcher.

Non-Tujuan Fase Awal

  • Membangun kernel sendiri sebagai default.
  • Mengganti stock Android/u-boot internal eMMC.
  • Full secure boot chain.
  • OTA delta kompleks seperti casync/RAUC penuh.
  • Optimasi ukuran image ekstrem.
  • Support semua model Amlogic di luar S905X.

6.1 Prinsip Arsitektur

  • Boot proven first: kernel dan DTB dari Armbian menjadi default sampai kernel custom terbukti.
  • Systemd native: service UOS dibuat sebagai systemd units.
  • Language-agnostic: Rust tetap boleh, tetapi service boleh Go/Python/Node/C sesuai dependency.
  • Hardware compatibility over purity: stabil di HG680P/B860H lebih penting daripada OS from-scratch.
  • CI reproducible: semua modifikasi rootfs dilakukan scriptable di GitHub Actions.

7. Boot Flow

7.1 Boot Flow Target

Android reboot update
  ↓
Stock Amlogic u-boot update mode
  ↓
FAT32: aml_autoscript
  ↓
FAT32: s905_autoscript
  ↓
Fallback uEnv.txt (tanpa u-boot.ext)
  ↓
fatload zImage/vmlinuz + uInitrd + DTB
  ↓
booti
  ↓
Armbian/Debian userspace
  ↓
systemd target
  ↓
UOS TV shell

7.2 Ketentuan Boot Partition

  • Sector 0 berisi MBR.
  • Sector 1–4095 dikosongkan untuk menghindari FIP/mainline u-boot conflict.
  • Partition 1 FAT32 dimulai di LBA 4096.
  • Tidak boleh ada u-boot.ext di boot partition untuk HG680P/B860H awal, karena sebelumnya menyebabkan chainload mainline u-boot gagal.

7.3 Contoh uEnv.txt

LINUX=/zImage
INITRD=/uInitrd
FDT=/dtb/amlogic/meson-gxl-s905x-p212.dtb
APPEND=root=/dev/mmcblk1p2 rootwait rw rootfstype=ext4 console=ttyAML0,115200n8 console=tty0 no_console_suspend consoleblank=0 net.ifnames=0
Catatan: root=/dev/mmcblk1p2 dipakai karena flow sebelumnya memakai MBR disk signature 0x00000000. Di fase berikutnya sebaiknya beralih ke UUID/PARTUUID yang stabil bila memungkinkan.

12.2 Dua Model Build

Model A — Modify Existing Armbian Image

Download image Armbian yang sudah proven, mount, inject UOS.

Kelebihan: paling cepat bootable.

Kekurangan: image lebih besar, kontrol base lebih rendah.

Model B — Build RootFS Debian Minimal

Debootstrap rootfs arm64, lalu copy kernel/boot dari ophub.

Kelebihan: lebih bersih dan reproducible.

Kekurangan: butuh setup chroot lebih banyak.

Keputusan implementasi: gunakan Model A — Modify Existing Armbian Image terlebih dahulu. Prioritas awal adalah image harus boot di hardware nyata dan berhasil load browser/UI shell. Setelah boot + browser stabil, baru lakukan cleanup/minimization dengan menghapus file, paket, service, kernel artifact, dan dokumentasi Armbian yang tidak diperlukan.

12.2.1 Strategi Model A Bertahap

FaseFokusYang DilakukanYang Belum Dilakukan
Fase A1 — Boot First Validasi boot hardware. Download image Armbian proven, inject UOS minimal, pertahankan kernel/DTB/initrd/modules asli Armbian. Belum hapus paket/file besar; belum optimasi ukuran.
Fase A2 — Browser First UI shell muncul otomatis. Install/aktifkan Cog/WPE atau Chromium kiosk, copy UI assets, buat systemd launcher. Belum hardening penuh; browser runtime boleh masih besar.
Fase A3 — Hardware Basics Input, network, WiFi, audio. Test remote/keyboard, NetworkManager, ALSA/PipeWire, firmware tambahan. Belum remove debug tools yang masih dibutuhkan troubleshooting.
Fase A4 — Cleanup/Minimize Kecilkan image setelah stabil. Remove package/file/service yang tidak perlu, clean apt cache/logs/docs, disable Armbian first-run extras. Tidak mengubah boot chain yang sudah proven kecuali ada alasan kuat.

12.2.2 Cleanup Setelah Browser Berhasil

Cleanup hanya dilakukan setelah acceptance berikut terpenuhi: image boot di HG680P/B860H, systemd mencapai multi-user target, browser/UI tampil otomatis, dan input dasar berfungsi.

  • Hapus apt cache: /var/cache/apt/archives/*.deb.
  • Hapus log build dan journal lama: /var/log/* dengan pengecualian struktur direktori yang dibutuhkan.
  • Reset machine identity: /etc/machine-id dikosongkan untuk regenerate saat first boot.
  • Hapus dokumentasi/manpage/locales yang tidak dibutuhkan bila ukuran image menjadi concern.
  • Disable service Armbian yang tidak relevan, misalnya first-run wizard bila tidak dipakai.
  • Hapus package development: compiler, headers, debug tools berat — kecuali untuk image debug.
  • Pertahankan kernel, initrd, DTB, firmware, dan modules yang sudah terbukti boot sampai ada pengganti yang proven.
Guardrail: cleanup tidak boleh dilakukan sebelum boot + browser proven. Jika cleanup menyebabkan regresi, workflow harus bisa build varian debug-full yang masih berisi tools lengkap untuk troubleshooting.

12.3 Cache Strategy CI/CD

Requirement penting: workflow tidak boleh rebuild/download ulang komponen besar bila input komponen tersebut tidak berubah. Cache harus dipisah per domain agar perubahan UI/service tidak membatalkan cache kernel, Armbian base, rootfs package, atau toolchain.

12.3.1 Prinsip Cache

  • Cache key harus berbasis input relevan, bukan seluruh repo. Contoh: cache kernel hanya berubah bila versi kernel, URL kernel, patch kernel, atau config kernel berubah.
  • Pisahkan cache mahal dan cache cepat. Armbian base image, kernel/DTB/initrd, apt package cache, Rust/Go/Node dependencies, dan browser runtime harus punya key terpisah.
  • Gunakan restore-key fallback. Jika exact key miss, workflow boleh restore cache versi dekat lalu hanya update delta.
  • Jangan cache output final image sebagai dependency utama. Image final boleh artifact, tapi cache sebaiknya menyimpan base/rootfs layer yang reusable.
  • Selalu ada step prepare/normalize setelah restore cache. Jika struktur cache lama berubah, step ini memperbaiki symlink/nama file tanpa download ulang.
  • Cache invalidation harus eksplisit. Parameter nocache=true hanya dipakai saat benar-benar perlu, bukan default.

12.3.2 Cache Domain yang Wajib

Cache DomainIsiKey InputHarus Invalid Bila
Armbian base image Downloaded/extracted Armbian image atau rootfs base. armbian-base-${ARMBIAN_IMAGE_VERSION}-${IMAGE_SHA256} Versi image/URL/checksum berubah.
Ophub kernel bundle zImage/vmlinuz, uInitrd, dtb/amlogic/*.dtb, optional modules. ophub-kernel-${OPHUB_KERNEL_VERSION}-${KERNEL_ASSET_SHA256}-vN Versi kernel, asset checksum, atau struktur prepare berubah.
Apt package cache /var/cache/apt/archives atau local apt proxy cache. apt-${DEBIAN_CODENAME}-${hash(package-list.txt)} Codename atau daftar paket berubah.
Rust build cache Cargo registry, git deps, target incremental per target triple. rust-${target}-${hash(Cargo.lock)}-${hash(relevant Cargo.toml)} Cargo.lock atau crate terkait berubah.
Node/UI cache node_modules atau package manager store. node-${hash(package-lock/pnpm-lock/yarn.lock)} Lockfile UI berubah.
Browser runtime cache Prebuilt Cog/WPE/Chromium package bila tidak dari apt repo biasa. browser-${runtime}-${version}-${sha256} Runtime/version berubah.
WiFi firmware/driver overlay Firmware blobs dan optional out-of-tree module artifact. wifi-${kernel_version}-${hash(scripts/build-s905x-wifi.sh)}-${hash(firmware-manifest)} Kernel ABI, firmware manifest, atau build script berubah.

12.3.3 Contoh Cache Key yang Direkomendasikan

env:
  ARMBIAN_IMAGE_VERSION: "26.05.0-s905x-b860h-6.12.91"
  OPHUB_KERNEL_VERSION: "6.12.91"
  DEBIAN_CODENAME: "trixie"

# Kernel cache
key: ophub-kernel-${{ env.OPHUB_KERNEL_VERSION }}-${{ hashFiles('configs/kernel/ophub-kernel.lock') }}-v2
restore-keys: |
  ophub-kernel-${{ env.OPHUB_KERNEL_VERSION }}-
  ophub-kernel-

# Apt/rootfs package cache
key: apt-${{ env.DEBIAN_CODENAME }}-${{ hashFiles('configs/armbian/packages.txt') }}
restore-keys: |
  apt-${{ env.DEBIAN_CODENAME }}-

# Rust cache
key: rust-aarch64-${{ hashFiles('Cargo.lock', 'crates/**/Cargo.toml') }}

12.3.4 Cache Hit Observability

Setiap workflow run wajib menampilkan ringkasan cache di log dan artifact metadata:

Cache summary:
- armbian-base: hit/miss, key=...
- ophub-kernel: hit/miss, key=...
- apt-packages: hit/miss, key=...
- rust: hit/miss, key=...
- ui: hit/miss, key=...
- wifi: hit/miss, key=...

File manifest.json artifact harus mencatat cache key, source URL, SHA256, versi kernel, versi base image, dan commit SHA agar build bisa diaudit.

12.3.5 Acceptance Criteria Cache

  • Jika hanya file UI berubah, cache Armbian base, kernel, apt package, dan WiFi tidak boleh invalid.
  • Jika hanya service Rust tertentu berubah, cache Armbian base dan kernel tetap hit; Rust dependency cache tetap hit kecuali Cargo.lock berubah.
  • Jika hanya uEnv.txt atau boot script berubah, kernel bundle tidak didownload ulang.
  • Jika build rerun di commit yang sama dengan nocache=false, semua cache domain besar harus hit.
  • Waktu build incremental target MVP maksimal 15–25 menit, bukan rebuild 60–120 menit.
  • Perubahan pada script prepare cache harus memakai suffix versi seperti -v2, bukan mengandalkan cache lama yang strukturnya incompatible.

13. Update dan OTA

13.1 MVP Update

  • Manual image replacement via SD card.
  • App/service update via package tarball atau git release artifact.
  • Rootfs tetap writable untuk development.

13.2 Fase Lanjut

  • A/B rootfs partition atau RAUC.
  • Signed update manifest.
  • Rollback otomatis bila boot gagal.
  • Delta update untuk menghemat bandwidth.

14. Security dan Hardening

14.1 MVP Security

  • Disable password root default atau paksa ganti password saat first boot.
  • SSH optional: enabled hanya development image, disabled production image.
  • Service berjalan sebagai user non-root bila memungkinkan.
  • Firewall basic via nftables/ufw bila network exposure ada.
  • Update channel ditandatangani untuk fase OTA.

14.2 Production Hardening

  • Read-only system partition.
  • Writable overlay untuk config/logs.
  • Least privilege systemd sandboxing: NoNewPrivileges, ProtectSystem, PrivateTmp.
  • Audit dependency browser/runtime.

19.3 Root Cause yang Ditemukan di CI

Root cause: Armbian first-login wizard terpicu oleh CI sendiri karena script menggunakan chroot ... /bin/bash -lc. Opsi -l menjalankan login shell, sehingga /etc/profile.d/armbian-check-first-login.sh dieksekusi dan memanggil /usr/lib/armbian/armbian-firstlogin selama file /root/.not_logged_in_yet masih ada.

Gejala yang muncul:

Web Configuration Available!
Connect to Wi-Fi SSID: armbian-armbiansetup
Then open http://10.42.0.1 in your browser
Or continue with CLI setup below...
stty: standard input: Inappropriate ioctl for device
Warning: Weak password, it is WAY too short!
read @issues

File pemicu di rootfs Armbian:

/etc/profile.d/armbian-check-first-login.sh
/root/.not_logged_in_yet
/usr/lib/armbian/armbian-firstlogin

Isi logic pemicu:

if [ -w /root/ -a -f /root/.not_logged_in_yet ]; then
    bash /usr/lib/armbian/armbian-firstlogin
fi

19.4 Fix yang Sudah Diterapkan

  • Hapus marker first-login sebelum menjalankan chroot apa pun:
/root/.not_logged_in_yet
/etc/armbian_first_run.txt
/boot/armbian_first_run.txt
/etc/profile.d/armbian-check-first-login.sh
/etc/profile.d/armbian-check-first-login-reboot.sh
  • Mengganti semua chroot command dari:
chroot "$ROOT_MNT" /bin/bash -lc "..."

menjadi:

chroot "$ROOT_MNT" /bin/bash -c "..."
  • Menambahkan disable/mask first-run related services.
  • Menambahkan noninteractive apt install agar package install tidak menunggu input.
  • Workflow default shell dipaksa bash.

20. Rekomendasi Implementasi Pertama

Langkah paling pragmatis: buat CI baru atau mode baru di workflow existing bernama armbian-base yang memodifikasi image Armbian proven, bukan membangun Debian rootfs dari nol.
  1. Download image Armbian S905X proven.
  2. Mount partition boot dan rootfs di CI.
  3. Pertahankan boot files/kernel/modules/DTB asli Armbian.
  4. Copy UOS UI assets dan service binaries/scripts minimal.
  5. Install systemd units untuk auto-start browser/UI shell.
  6. Set hostname, user, auto-start target.
  7. Build artifact debug-full terlebih dahulu tanpa cleanup agresif.
  8. Test boot hardware sampai browser tampil.
  9. Setelah proven, aktifkan tahap cleanup/minimize bertahap.
  10. Unmount, fsck, upload image final.

Pendekatan ini menjaga keunggulan utama: semua file boot/kernel/modules tetap berasal dari image Armbian yang sudah berhasil di hardware nyata. Cleanup dilakukan belakangan agar tidak menambah variabel saat debugging boot/browser.

Dokumen dibuat untuk proyek UOS TV. Fokus: migrasi basis OS ke Armbian/Debian agar boot dan integrasi hardware lebih cepat stabil pada Amlogic S905X STB.
